## 1. Potential bugs / correctness risks

### 1.1. Alias resolution and device identity corner cases

- The production pipeline relies on `BatteryReaderOrchestrator` plus `PhysicalDeviceIdentityResolver` and `ThresholdSettings` alias maps to merge readings and apply per-device settings.
- Risk: if the alias resolution pipeline fails to resolve a renamed device, thresholds and suppression settings silently fall back to defaults, which can surprise the user.
- Risk: fuzzy matching in the alias pipeline can incorrectly associate readings from a different device with an existing alias, potentially mis-labelling devices in the UI.

**Impact**

- Misapplied thresholds (false positives or missed alerts).
- Confusing device naming and history in the UI.

**Mitigation**

- Treat low-confidence alias matches as “suggestions only” that never affect alerting until explicitly confirmed in the options UI.
- Persist both canonical identity and last-seen raw identity, and surface discrepancies directly in the scan window for user inspection.
- Extend tests around alias collision cases and near-duplicate names.

### 1.2. Race conditions around `_lastKnown` and UI updates

- `BluetoothBatteryMonitor`, `Scanner`, and `PollingOrchestrator` all interact with a shared `_lastKnown` map of devices.
- The architecture doc states that manual scans and background polls share this state to avoid interleaving.
- If any accesses to `_lastKnown` are not guarded consistently, there is a risk of:
  - Transient `KeyNotFoundException` if a device is removed while processing.
  - UI showing partially updated lists.

**Mitigation**

- Ensure all mutations of `_lastKnown` are channelled through a single orchestrator and protected consistently (lock, immutable snapshots, or dedicated actor).
- Consider snapshotting the dictionary to an immutable collection for each UI update.

### 1.3. Timer/async shutdown ordering

- `BluetoothBatteryMonitor` owns a `System.Threading.Timer`, a `CancellationTokenSource`, and a `TaskTracker`.
- Incorrect disposal ordering can lead to:
  - Timer callbacks firing after disposal started.
  - Tasks awaiting UI-bound operations after the tray app has already shut down.

**Mitigation**

- Enforce disposal order strictly: stop timer → cancel CTS → await `TaskTracker` → dispose readers.
- Add tests that simulate rapid create/dispose cycles.

### 1.4. Classic reader dependence on Windows device properties

- Classic path uses SetupAPI/WMI properties to read `DEVPKEY_Bluetooth_Battery`.
- Devices that only populate this property when “actively connected” can cause intermittent zero/`null` readings.

**Mitigation**

- Treat missing/zero values as “no reading” rather than “0 %”, and ensure hysteresis logic never transitions to `Low` purely on such non-readings.
- Extend tests to cover missing Classic readings while GATT remains healthy.

### 1.5. Laptop battery monitoring vs. Windows power plans

- `LaptopBatteryMonitor` polls `PowerStatus` plus `Win32_Battery` and runs its own threshold logic.
- On some hardware, `Win32_Battery` is absent or stale, which can skew results.

**Mitigation**

- Prefer `PowerStatus` as the primary source; treat `Win32_Battery` as optional enrichment.
- Add explicit detection and a log line when `Win32_Battery` appears inconsistent so support can correlate reports.

### 1.6. ntfy push integration edge conditions

- ADR-014 and related tests show support for remote notifications via ntfy.
- Risk: network failures, invalid topics, and HTTP timeouts could block or delay the main polling loop if not isolated properly.

**Mitigation**

- Ensure ntfy publishing is always fire-and-forget from the perspective of the main polling path (e.g. queue + background sender) and never awaited on the hot loop.
- Enforce strict per-request timeouts and circuit-breaking on ntfy failures.

## 2. Code smells and maintainability issues

### 2.1. Large multi-responsibility classes in UI layer

- `OptionsForm`, `ScanWindow`, and `TrayApp` are relatively large classes with multiple responsibilities: UI layout, validation, orchestration, and direct manipulation of settings objects.

**Issues**

- Difficult to reason about and test new behaviours.
- Risk of subtle bugs when new options are added without updating all dependent logic.

**Refactor direction**

- Extract passive view models for device lists, threshold editing, and notifications preferences.
- Keep WinForms code-behind focused only on control wiring and data binding.

### 2.2. Tight coupling between settings and domain logic

- `ThresholdSettings` currently owns persistence, domain rules, and raises change events used directly by UI and monitors.

**Issues**

- Violates strict single-responsibility separation between domain configuration and persistence.
- Makes it hard to evolve the settings file format or introduce alternative storage.

**Refactor direction**

- Maintain a pure domain `ThresholdSettings` model independent of file I/O.
- Move JSON handling and atomic writes into a dedicated persistence service that maps to/from the domain model.

### 2.3. Legacy `IBatteryReader` path still present

- `DeviceAggregationPipeline` and `IBatteryReader` remain to support older tests and constructors, but are no longer used in the production wiring.

**Issues**

- Increases cognitive load for newcomers.
- Risk of future regressions if someone reuses these types in new code instead of the orchestrator.

**Refactor direction**

- Mark legacy types as `[Obsolete]` with clear comments.
- Plan a phased removal once tests depending on the pipeline are migrated to the orchestrator path.

### 2.4. Mixed concerns in `BatteryReaderOrchestrator`

- The orchestrator class currently handles:
  - Concurrency orchestration.
  - Device-class filtering.
  - Alias resolution.
  - Discovery logging.

**Issues**

- Class growth risk as new ADRs are implemented.

**Refactor direction**

- Extract dedicated collaborators for alias resolution, filtering, and logging.
- Keep the orchestrator focused on “fan-out, merge, classify”.

### 2.5. Manual thread marshaling scattered in UI glue code

- `ScanCoordinator` and related types perform manual `BeginInvoke`/`SynchronizationContext` operations.

**Issues**

- Call-site level `BeginInvoke` leads to repetitive boilerplate and increased risk of forgetting marshaling.

**Refactor direction**

- Centralise UI dispatching in a small helper/adapter and use it consistently from all background consumers.

## 3. Adherence to design decisions and fundamental docs

### 3.1. ADR-001: Single process, manual DI

- `Program.cs` is the composition root and manually wires all dependencies without any external DI container.

Assessment: **Design adhered to.**

### 3.2. ADR-002: Dual Bluetooth reader strategy

- Both GATT and Classic readers are implemented and wired through `BatteryReaderOrchestrator`.
- Merging logic favours GATT when duplicate identities are observed.

Assessment: **Design adhered to.**

### 3.3. ADR-003: Polling over push

- Background monitoring is implemented using a 60 s `System.Threading.Timer`.
- No attempt is made to receive system push events for battery changes.

Assessment: **Design adhered to.**

### 3.4. ADR-004: Threshold hysteresis

- `PollingOrchestrator` implements a `BatteryAlertState` FSM with hysteresis so that small oscillations around thresholds do not cause repeated notifications.
- Tests cover classification and eviction behaviours.

Assessment: **Design adhered to.**

### 3.5. ADR-005: Atomic settings persistence

- Settings are written to a temporary file then moved into place to avoid corruption.
- Corrupt or missing files reset to defaults and do not crash the process.

Assessment: **Design adhered to.**

### 3.6. ADR-006: WinRT Toast notifications

- `NotificationService` wraps `ToastNotificationManager` and registers an AUMID.
- Click handling routes back to the scan UI as described.

Assessment: **Design adhered to.**

### 3.7. ADR-007: TaskTracker cooperative shutdown

- `TaskTracker` exists and is used from `BluetoothBatteryMonitor` to track background tasks and coordinate shutdown.

Assessment: **Design adhered to**, with the caveat that tests for aggressive shutdown scenarios could be extended.

### 3.8. ADR-008: Options record for constructor parameter groups

- `BluetoothMonitoringInfrastructure` and related types encapsulate parameter groups in records instead of high-arity constructors.

Assessment: **Design adhered to.**

### 3.9. ADR-009 to ADR-013: Device identity, events, and settings SRP

- Identity resolution, separate events for general vs laptop settings, and separation of settings domain vs persistence are all reflected in the code structure.

Assessment: **Mostly adhered to**, with some residual coupling as described in section 2.2.

### 3.10. ADR-014 to ADR-019: ntfy, alias migration, filtering, enumeration, discovery logging, deep scan

- ntfy integration, alias migration, device-class filtering, passive enumeration, structured logging, and deep scan UX all appear to be implemented at least partially, with corresponding tests.

Assessment: **Design mostly adhered to**, but operational robustness for ntfy and deep scan cancellation should be hardened.

## 4. Missing or weakly supported features

### 4.1. Telemetry and diagnostics for end users

- There is structured discovery logging, but no obvious user-facing diagnostics panel.

**Impact**

- Difficult for non-technical users to understand why a device does not appear or why thresholds are not being applied.

**Recommendation**

- Add a simple diagnostics view in the options dialog or scan window showing:
  - Last scan time and result counts per reader.
  - Per-device last-reading source (GATT/Classic) and reliability score.
  - Alias mappings and any unresolved candidates.

### 4.2. Multiple adapter awareness

- README explicitly states multiple Bluetooth adapters are “not explicitly tested”.

**Recommendation**

- Either:
  - Explicitly support adapter selection in settings, or
  - Detect multiple adapters and log a clear warning in diagnostics and documentation.

### 4.3. Advanced scheduling / power-awareness

- Polling interval is fixed at 60 s, with some power-mode awareness.

**Recommendation**

- Introduce adaptive polling based on:
  - Laptop on AC vs battery.
  - Recent activity (e.g. no devices seen for N polls → back off).

### 4.4. Policy for flaky devices

- Some devices will regularly disappear/reappear or report wildly fluctuating battery.

**Recommendation**

- Add per-device stability heuristics and UI-level indication when a device is considered “flaky”, to prevent user confusion.

---

## 5. Prioritised improvement backlog

1. Harden alias resolution and identity mapping to eliminate silent mis-association of devices.
2. Simplify and modularise `BatteryReaderOrchestrator` and UI forms to keep growth manageable.
3. Tighten shutdown semantics and tests around timers and `TaskTracker` to avoid edge-case hangs.
4. Introduce minimal diagnostics UI to make device and reader behaviour observable without attaching a debugger.
5. Plan deprecation and removal of legacy `IBatteryReader`+`DeviceAggregationPipeline` path once tests are migrated.
