# ADR-014 â€” Remote Mobile Notifications via ntfy.sh Public Server

## Status
Proposed

## Context
BTChargeTrayWatcher currently delivers battery threshold alerts exclusively as Windows desktop toast notifications. This is sufficient when the user is at their PC, but a common usage scenario is that Bluetooth devices are charging or discharging while the user is away from the desk. In that case all alerts are missed.

Users reliably carry a mobile phone. A lightweight push notification channel to the phone would close this gap without requiring server infrastructure.

## Decision
Adopt **ntfy.sh** as the remote notification channel:

- Use the **public `ntfy.sh` server** â€” no self-hosted server, no account, no additional infrastructure required.
- The Windows app publishes battery alert messages via HTTP POST to `https://ntfy.sh/{topic}`.
- The user subscribes to the same topic in the ntfy mobile app (Android or iOS).
- The topic name is treated as a **shared secret**: it must be randomly generated and sufficiently long to prevent enumeration.
- Local Windows toast notifications continue to operate unchanged. ntfy is an additive second channel.

## Constraints
- No extra infrastructure may be introduced.
- No sensitive user data (MAC addresses, Windows usernames, machine identifiers) may be included in notification payloads.
- Battery percentage and device display name are acceptable payload content.
- The topic must be randomly generated by the app; the user must not be expected to choose one.
- The local notification path must remain fully operational regardless of ntfy configuration state.

## Architecture
- Introduce `INotificationChannel` with implementations `WindowsToastNotificationChannel` and `NtfyNotificationChannel`.
- Fan out through a `NotificationDispatcher` that iterates enabled channels.
- `NtfyNotificationChannel` owns the HTTP transport concern exclusively.
- `NtfyIntegrationSettings` is a value object stored in `ThresholdSettings` and persisted via `SettingsPersistence`.
- `ThresholdSettings` exposes configuration state only; no transport logic enters the domain model.

## Consequences
- **Positive**: Zero infrastructure cost; near-instant delivery on Android and iOS; simple onboarding via the tray menu.
- **Negative**: Dependency on the public ntfy.sh service; public topics are accessible to anyone who knows the topic name (mitigated by random generation).
- **Neutral**: iOS instant delivery relies on ntfy.sh forwarding to APNs, which is the standard behavior of the public server.
