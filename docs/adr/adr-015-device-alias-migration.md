**Status:** Proposed
**Date:** 2026-05-20

## Context

`ADR-009` defines a clear split between runtime identity (`DeviceId`) and user-facing configuration keyed by `Name`. A recurring usability gap is that when a device is re-paired, renamed, or provisioned with a slightly different display name, the user's per-device threshold overrides and ignore lists silently stop applying. This leads to confusion: users expect their aliases and overrides to "follow" the physical device even when Windows exposes a different ephemeral `DeviceId` or minor name variation.

We want improved recognition while respecting the project's constraints:
- No exposing GUIDs in the UI; configuration remains keyed by `Name`.
- Persistence must remain via `ThresholdSettings` + `SettingsPersistence` (ADR-013 / ADR-005).
- Any automatic mapping must be conservative to avoid misapplying user preferences.

## Decision

1. Add an alias/history surface to the domain `ThresholdSettings`: a lightweight mapping of historical display-name variants to a single *canonical* display name. Conceptually:

   - `AliasMap: Dictionary<string /* normalized old name */, string /* canonical name */>`

   This map is part of the domain snapshot and is persisted by `SettingsPersistence` as part of the existing JSON settings DTO.

2. Implement a *non-destructive* heuristic pipeline in `DeviceAggregationPipeline` that proposes and applies name resolution in the following precedence (high â†’ low):

   - Exact case-insensitive name equality.
   - `AliasMap` lookup by a normalized name key.
   - Normalised name equivalence (trim, punctuation removal, collapse whitespace).
   - High-confidence fuzzy match (Levenshtein similarity >= 0.85) only when no other candidate exists.

3. Auto-apply mappings only when the pipeline yields *high confidence* (exact or alias lookup). For fuzzy matches, only surface a *suggested mapping* in the UI (ScanWindow / OptionsForm) requiring the user's explicit confirmation before persisting to `AliasMap`.

4. Expose an explicit UI surface under `OptionsForm` (Devices tab) where the user can:

   - View suggested mappings generated by the last scan.
   - Confirm or reject individual suggestions.
   - Manually edit alias mappings (advanced users).

5. During software upgrades, no automated remapping happens as part of startup; suggested matches are produced lazily at scan time. If an administrator or user chooses, a one-time migration tool may be offered later to bulk-apply high-confidence mappings.

## Rationale

- Keeps the configuration model (`ThresholdSettings`) the single source of truth for persisted user preferences (ADR-013).
- Avoids surfacing GUIDs to users while still making aliases resilient to re-pairing/name variants.
- Conservative auto-application reduces the risk of silently assigning settings to the wrong device.

## Implementation Notes

- Normalization must be deterministic and locale-insensitive: lower-case, remove diacritics, strip punctuation, collapse whitespace.
- Levenshtein or similar string metric is acceptable; implementations must be efficient and limited to candidates produced by the scan (do not compare against all historical names for large growth scenarios).
- Suggested mappings should carry a confidence score and the originating scan timestamp so the UI can explain why a suggestion was made.

## Consequences

- Small increase in settings file size due to `AliasMap` entries.
- Possible user confusion if false-positive fuzzy matches are suggested; mitigated by explicit confirmation requirement.
- Additional UI surface required in `OptionsForm` and a small changeset to `ThresholdSettings` and `SettingsPersistence`.
